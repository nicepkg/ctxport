# Context 复制场景交互痛点分析报告

> 版本：v1.0 | 日期：2026-02-07
> 方法论：Alan Cooper Goal-Directed Design
> 分析视角：交互设计总监
> 基于：现有 Persona 调研 + 社区信号 + 竞品分析 + Web 调研

---

## 1. 补充 Persona 定义

现有 Persona 体系中，Alex（Primary）、Maya（Secondary）和 Sam（Supplementary）覆盖了核心用户群。但在深入分析 context 复制场景后，我发现三个被忽略的行为模式需要独立的 Persona 来表达。

### Persona 4: Leo — The Team Lead Curator

> "我的团队六个人，每个人都在自己的 AI 里讨论方案。每周 Review 时我需要把大家的 AI 对话拼在一起看全貌，这比写代码还累。"

| 属性 | 描述 |
|------|------|
| 年龄 | 32-42 |
| 角色 | 技术 Lead / 工程经理 / 架构师 |
| 技术水平 | 高级，10+ 年经验 |
| 工作模式 | 管理 3-8 人团队，自己也写代码 |
| AI 工具使用 | 日均 2-3 小时，偏重审查和架构决策 |

**行为模式**

- 要求团队成员将 AI 讨论结果以"会议纪要"形式整理到 Confluence/Notion
- 自己在 Claude 中做架构 Review，需要看到团队成员之前与 AI 的讨论上下文
- 经常需要"把 A 同事在 ChatGPT 中的方案"和"B 同事在 Claude 中的实现"放在一起比较
- 用 AI 做 Code Review 时，需要携带原始的需求讨论上下文

**核心 Context 复制流程**

```
[1. 收集]  从团队成员那里获取 AI 对话（Slack 链接/截图/手动转发）
     | (10-20 min)
[2. 整理]  将多个对话片段合并整理为连贯的上下文
     | (15-30 min)
[3. 投喂]  粘贴到自己的 AI 工具中进行分析/Review
     | (5 min)
[4. 输出]  将 Review 结果分享回团队
```

**挫折点**

1. **团队 AI 对话不可见**：不知道团队成员和 AI 讨论了什么，也无法审查 AI 给出的建议质量
2. **合并多人上下文极其痛苦**：3 个人的 AI 对话，格式不同、工具不同、上下文断裂
3. **分享链路断裂**：ChatGPT 的共享链接只读且有有效期，Claude 无原生分享
4. **缺乏审计能力**：无法追溯"这段代码是哪次 AI 对话中产生的决策"

**与 Alex 的区别**：Alex 的流程是"我 -> AI -> 我 -> 另一个 AI"（个人闭环），Leo 的流程是"团队成员的 AI -> 我的 AI -> 团队"（多人协作闭环）。他需要的不仅是复制自己的会话，还需要**汇聚和比较**多个来源的 AI 上下文。

### Persona 5: Kai — The Context Window Firefighter

> "每天至少有 3 次 context window 爆掉的情况。我已经练出了一套'紧急抢救上下文'的肌肉记忆，但每次还是很痛苦。"

| 属性 | 描述 |
|------|------|
| 年龄 | 24-35 |
| 角色 | 全栈开发者 / AI 重度用户 |
| 技术水平 | 中高级 |
| 工作模式 | 日均 50+ 次 AI 交互，深度依赖 AI 编码 |
| AI 工具使用 | 日均 6-8 小时，几乎所有工作都通过 AI 完成 |

**行为模式**

- 在 Cursor 中连续工作 2-3 小时后，context window 接近上限
- 已建立"context checkpoint"习惯：每 30 分钟手动总结一次当前状态
- 使用 CLAUDE.md 作为"项目记忆"，但手动维护成本高
- 经常需要在 context window 爆掉前"抢救"关键上下文片段到新会话
- 使用多个 AI 不是因为偏好，而是因为单个工具的 rate limit 和 context 限制

**核心 Context 复制流程**

```
[1. 感知危机]  AI 响应质量下降 / 看到 token 计数接近上限
     | (即时)
[2. 紧急打包]  快速总结当前上下文的关键信息
     | (3-5 min, 高压)
[3. 开新会话]  在同一工具或另一个工具中开启新会话
     | (即时)
[4. 恢复上下文]  粘贴打包的上下文 + 项目背景
     | (5-10 min)
[5. 验证恢复]  测试 AI 是否理解了之前的上下文
     | (2-5 min)
```

**挫折点**

1. **Context Window 是黑盒**：Cursor 宣称 200K tokens 实际只有 70-120K，用户无法准确预判何时会爆
2. **"紧急抢救"的时间压力**：AI 响应质量已经在下降，但用户还在拼命总结上下文
3. **手动 compaction 质量不稳定**：匆忙总结的上下文经常遗漏关键信息
4. **恢复后的"冷启动"成本**：新会话需要 5-10 分钟才能达到之前会话的理解水平
5. **跨工具逃逸**：当一个工具 rate limit 了，被迫切换到另一个工具，上下文迁移更痛苦

**与 Alex 的区别**：Alex 的切换是**主动的**（"我想用更适合的工具"），Kai 的切换是**被迫的**（"这个工具的 context 爆了 / rate limit 了"）。Kai 的核心需求是**速度和可靠性**——在紧急情况下快速打包并恢复上下文。

### Persona 6: Dana — The AI Skeptic Adopter

> "我知道 AI 很有用，但每次复制粘贴的时候我都在想：这些数据会去哪？谁能看到？我的客户知道我在用 AI 吗？"

| 属性 | 描述 |
|------|------|
| 年龄 | 35-50 |
| 角色 | 企业开发者 / 咨询顾问 / 金融/医疗行业从业者 |
| 技术水平 | 中高级 |
| 工作模式 | 受合规约束，需要数据审计 |
| AI 工具使用 | 日均 1-2 小时，谨慎使用 |

**行为模式**

- 使用 AI 前会先脱敏数据（手动替换客户名、项目代号）
- 不使用 ChatGPT Memory 功能，担心数据泄露
- 复制 AI 输出到正式文档前，会手动审查和编辑
- 对 Anthropic 2025 年 ToS 变更（默认用会话数据训练模型）高度敏感
- 需要向客户/管理层证明 AI 使用的合规性

**核心 Context 复制流程**

```
[1. 脱敏准备]  手动替换敏感信息（客户名、数据、代码中的密钥）
     | (5-15 min)
[2. 投喂 AI]  将脱敏后的上下文粘贴到 AI
     | (即时)
[3. 获取输出]  AI 生成回复
     | (即时)
[4. 二次脱敏]  检查 AI 输出中是否无意间推断出了敏感信息
     | (3-5 min)
[5. 复制输出]  将清洁的输出复制到目标文档
```

**挫折点**

1. **手动脱敏耗时且不可靠**：容易遗漏变量名、注释中的信息、日志中的 IP 地址
2. **无法验证"我的数据有没有被训练"**：Anthropic/OpenAI 的数据使用政策模糊
3. **跨平台复制放大隐私风险**：从 A 平台复制到 B 平台，数据暴露面加倍
4. **合规审计无记录**：无法证明"哪些数据被发送给了哪个 AI"

**设计启示**：对 Dana 来说，CtxPort 的"本地处理"不是功能卖点——它是**使用前提**。如果 CtxPort 的 context 复制过程涉及任何远程服务器，Dana 不会使用。

## 2. 每个 Persona 的 Context 复制流程与痛点映射

### 2.1 Alex（Primary）— 典型流程与痛点

| 步骤 | 当前行为 | 痛点 | 严重度 | CtxPort MVP 是否覆盖 |
|------|---------|------|--------|---------------------|
| 发现需要迁移 | 在 ChatGPT 中完成讨论，准备切换工具 | 无提示，全靠用户主动意识 | -- | -- |
| 定位会话 | 在列表中滚动找之前的对话 | 无搜索/标签，列表冗长 | ★★★★★ | 部分（列表复制） |
| 打开会话 | 点击打开，等待加载 | 长会话加载 3-10 秒 | ★★★★☆ | 已解决（流程 2 不打开就复制） |
| 选择内容 | 全选 or 手动选择片段 | 无法只选关键部分 | ★★★☆☆ | 部分（右键菜单格式选项） |
| 复制 | Ctrl+C 或扩展按钮 | 格式丢失（代码缩进、表格） | ★★★★★ | 已解决（Markdown Bundle） |
| 切换工具 | Cmd+Tab / 新 Tab | 工具间无语义连接 | ★★☆☆☆ | 不在范围内 |
| 粘贴 | Cmd+V | 目标工具不理解 Bundle 结构 | ★★★★☆ | 部分（格式保真，但无"导入"端） |
| 补充上下文 | 手动添加项目背景说明 | 重复劳动，每次 3-10 分钟 | ★★★★☆ | 未覆盖 |

**Alex 的未覆盖痛点**：
- **"部分选择"需求**：Alex 不总是需要完整会话，经常只需要"最后 5 轮对话"或"所有代码块 + 最终结论"
- **"项目背景自动附加"**：每次复制都需要手动附加项目背景 prompt
- **"粘贴端适配"**：复制的 Bundle 粘贴到 Cursor 和 Claude Code 时，没有工具理解其结构

### 2.2 Maya（Vibecoder）— 典型流程与痛点

| 步骤 | 当前行为 | 痛点 | 严重度 | CtxPort MVP 是否覆盖 |
|------|---------|------|--------|---------------------|
| 发现需要迁移 | Context window 爆掉，被迫开新会话 | 恐慌，不知道丢了什么 | ★★★★★ | 未覆盖 |
| 理解"该复制什么" | 不知道哪些信息是关键上下文 | 缺乏技术判断力 | ★★★★★ | 未覆盖 |
| 操作复制 | 手动选择、Ctrl+C | 不确定是否复制完整 | ★★★☆☆ | 已解决（一键复制） |
| 理解 token 预算 | 不知道复制的内容有多大 | "200K tokens"无意义 | ★★★★☆ | 部分（显示 ~XK tokens） |
| 粘贴到新会话 | Cmd+V | 新 AI 是否理解了？不确定 | ★★★★☆ | 未覆盖 |
| 验证恢复 | 问 AI "你还记得我们之前讨论的吗？" | AI 可能假装记得 | ★★★★☆ | 未覆盖 |

**Maya 的未覆盖痛点**：
- **"智能摘要"需求**：Maya 无法判断什么是关键上下文，需要工具帮她提炼
- **"Token 可视化"需求**：不是数字而是视觉化——一个进度条显示"这些内容占目标工具 context window 的 40%"
- **"恢复验证"需求**：粘贴后如何确认新 AI 真的理解了？需要某种"上下文完整性检查"
- **"右键菜单"不可发现**：Maya 几乎不会右键点击，渐进式披露的高级功能她永远找不到

### 2.3 Leo（Team Lead）— 典型流程与痛点

| 步骤 | 当前行为 | 痛点 | 严重度 |
|------|---------|------|--------|
| 收集团队 AI 输出 | Slack 中要求团队成员转发 AI 对话截图 | 格式不一、信息丢失 | ★★★★★ |
| 合并多来源上下文 | 手动在 Google Doc 中拼接 | 极其耗时（30-60 min） | ★★★★★ |
| 投喂自己的 AI | 粘贴合并后的内容到 Claude | 往往超出 context window | ★★★★☆ |

**Leo 的未覆盖痛点**：
- **"收集他人会话"**：MVP 只支持复制自己正在浏览的会话，无法让团队成员"推送"会话
- **"多来源 Bundle 合并"**：即使团队成员各自用 CtxPort 复制了自己的会话，Leo 还是需要手动合并
- **"决策溯源"**：无法从最终代码追溯到"这个决策是哪次 AI 对话中做出的"

|------|---------|------|--------|
| 感知 context 即将爆 | AI 响应质量下降 or 看到 token 计数 | 没有提前预警 | ★★★★★ |
| 紧急打包 | 让 AI 总结当前上下文 | 高压下质量不稳定 | ★★★★☆ |
| 开新会话 | 在同一工具或切换工具 | 上下文断裂 | ★★★★★ |
| 恢复上下文 | 粘贴打包内容 + 项目背景 | "冷启动"需 5-10 分钟 | ★★★★☆ |
| 验证恢复质量 | 问测试问题 | AI 可能胡说 | ★★★☆☆ |

**Kai 的未覆盖痛点**：
- **"一键快照"需求**：在 context window 即将爆掉时，一键将当前会话状态打包为 Context Bundle
- **"增量复制"需求**：不需要复制完整会话，只需要"从上次快照到现在的增量"
- **"热恢复"需求**：粘贴 Bundle 后 AI 应该能"热启动"而非"冷启动"

## 3. 高频流程摩擦点分析

### 3.1 摩擦点 F1：批量操作流程的认知负担（严重度 ★★★★☆）

**当前设计**：进入批量模式 → 逐个勾选 → 点击"复制全部" → 等待 → 完成

**问题分析**：

1. **进入批量模式的门槛过高**：三种触发方式（快捷键 / 长按 / Popup）中，快捷键 `Cmd+Shift+E` 需要记忆，长按 500ms 无可发现性，Popup 按钮需要额外步骤。对 Maya（Vibecoder）来说，这三种方式都不直觉。

2. **选择过程缺乏"智能辅助"**：用户必须自己判断哪些会话属于同一个项目。在一个有 200+ 条会话的列表中，这个判断成本极高。Cooper 的交互礼仪原则："不要让用户做机器该做的事。"

3. **复制后的"黑盒感"**：用户勾选了 5 个会话点击"复制全部"，但不知道合并后的 Bundle 是什么样的。Toast 只显示"已复制 5 个会话（共 112 条消息）"——没有结构预览，没有顺序确认。

**改进建议**：

- **P1**：增加"智能分组建议"——基于会话标题和时间聚类，自动建议"这 5 个会话看起来属于同一项目，要一起复制吗？"
- **P2**：在批量复制完成后的 toast 中增加可展开的"Bundle 目录"——点击可看到合并后的结构大纲
- **P3**：为 Maya 增加一个更低门槛的入口——在 Extension Popup 中提供"最近项目快速打包"功能

---

### 3.2 摩擦点 F2：格式选择的不可发现性（严重度 ★★★☆☆）

**当前设计**：格式选项隐藏在右键菜单中（"完整会话"、"仅用户消息"、"仅代码块"、"精简版"）

**问题分析**：

1. **右键菜单对非开发者不可见**：Maya、Sam、Dana 几乎不会在一个小图标上右键点击。竞品 AI Chat Exporter v3.2.1 已经提供了明确的 checkbox 和 preset 选择器——更直观但更重。

2. **格式命名不够自解释**："精简版"是什么意思？"移除代码块内的注释和空行"——用户不理解这个定义，也无法预判效果。

3. **缺乏"格式预览"**：用户无法在复制前看到不同格式选项的效果差异。当前设计有一个可选的"预览面板"（点击 toast 中的"[预览]"链接），但这是复制**之后**的预览，不是**选择格式时**的预览。

**改进建议**：

- **P1**：在右键菜单的每个格式选项旁增加一行灰色说明文字（例如："仅代码块 — 提取所有 ``` 代码段，适合粘贴到 IDE"）
- **P2**：格式选项保持在右键菜单中（对 Alex 足够好），但在 Extension Popup 中为非开发者提供格式选择器（带简短说明和 token 估算）
- **P3**（长期）：右键菜单中增加实时预览——hover 格式选项时在侧边浮出 3 行内容预览

---

### 3.3 摩擦点 F3：复制后缺乏"目标端验证"（严重度 ★★★★☆）

**当前设计**：复制完成后显示 toast（"已复制 24 条消息 ~8.2K tokens"），用户切换到目标工具粘贴。

**问题分析**：

这是整个流程中最大的**信任黑洞**——用户复制了，但不知道粘贴后效果如何。

1. **格式保真无法验证**：用户在 ChatGPT 中看到格式正确的代码块，复制后粘贴到 Claude Code 中可能变成纯文本。没有工具告诉用户"复制的 Markdown 格式是否会在目标工具中正确渲染"。

2. **Token 估算的不确定性**：Toast 显示"~8.2K tokens"，但不同 AI 工具的 tokenizer 不同——ChatGPT 的 8.2K tokens 在 Claude 中可能是 9K。用户无法判断这个 Bundle 是否超出目标工具的 context window。

3. **上下文完整性不可验证**：复制的 24 条消息中，是否遗漏了关键的 system prompt 或 thinking 内容？用户粘贴到新 AI 后，AI 是否真的理解了？

4. **竞品对照**：Context Pack（context-pack.com）已推出 AI Memory Migration Platform，提供跨平台 context 迁移和验证——虽然方向不同（侧重 memory 而非 conversation），但表明市场认可"验证"环节的价值。

**改进建议**：

- **P1**：在 Toast 中增加"目标工具适配提示"——例如："已复制 ~8.2K tokens，适合 Claude (200K) / ChatGPT (128K) / Cursor (~100K)"，用颜色标识（绿=充裕，黄=刚好，红=可能超出）
- **P2**：提供一个简单的"剪贴板内容校验"功能——在 Extension Popup 中点击"查看剪贴板"可预览当前剪贴板中的 Context Bundle，确认格式和内容
- **P3**（长期）：在目标 AI 工具中自动检测粘贴的 Context Bundle 并提供"上下文加载确认"

---

### 3.4 摩擦点 F4：重复复制同一会话的不同部分（严重度 ★★★☆☆）

**当前设计**：每次复制都是"全量"操作——要么复制完整会话，要么通过格式选项做预设筛选。

**问题分析**：

1. **Alex 的实际需求**：很多时候 Alex 只需要"从第 15 条消息开始到最后"或者"这段代码块 + AI 的解释"——但现有设计没有消息级别的选择能力。

2. **竞品已有的能力**：AI Chat Exporter v3.2.1 提供了**逐条消息的 checkbox 选择**，允许用户精确选择要导出的消息。这说明市场已经认可"消息级别选择"的需求。

3. **"仅代码块"不够灵活**：有时候 Alex 需要"这段代码块 + 它前面的需求描述 + 它后面的解释"——这是一个三明治结构，纯粹的"仅代码块"无法满足。

**改进建议**：

- **P2**（非 MVP，但应纳入路线图）：增加"消息级别选择"模式——在会话详情页中，用户可以勾选特定消息，仅复制选中的消息
- **P3**（长期）：增加"范围选择"——类似 GitHub 的代码行范围链接，用户可以选择"从消息 #5 到 #12"

---

### 3.5 摩擦点 F5：快捷键冲突和可发现性不足（严重度 ★★☆☆☆）

**当前设计**：`Cmd+Shift+C` 复制完整会话，与 Chrome DevTools Console 快捷键冲突。

**问题分析**：

1. **冲突是真实的**：在 ChatGPT/Claude 页面上调试时，开发者经常需要打开 DevTools。`Cmd+Shift+C` 是 Element Inspector 的快捷键，被 CtxPort 覆盖后会导致开发者困惑。

2. **快捷键记忆成本**：4 种格式 x 4 个快捷键 + 批量模式 1 个 = 5 个快捷键需要记忆。对日常使用频率不高的格式选项来说，快捷键的 ROI 低。

3. **首次使用的引导不足**：当前设计在成功 toast 中附带快捷键提示"下次试试 Cmd+Shift+C 更快"——但只显示一次。用户可能在忙碌中忽略了这条提示。

**改进建议**：

- **P1**：将默认快捷键从 `Cmd+Shift+C` 改为不冲突的组合（例如 `Cmd+Shift+X`，X = eXport），或在检测到冲突时提供首次选择
- **P2**：减少快捷键数量——只保留"复制完整会话"和"进入批量模式"两个快捷键，其他格式用右键菜单即可

## 4. 跨平台交互断裂点分析

### 4.1 断裂点 B1：浏览器 -> CLI 工具（Claude Code / Cursor Terminal）

**断裂描述**：

用户在浏览器中用 CtxPort 复制了 Context Bundle 到剪贴板，切换到终端 `Cmd+V` 粘贴。

**断裂表现**：

- 终端的粘贴行为不同于图形界面——Markdown 格式在终端中是纯文本，无渲染
- Claude Code 接收到大段粘贴的文本时，可能触发安全检查或 token 限制
- 粘贴后的缩进可能被终端修改（特别是在 tmux/zsh 中）
- **没有"粘贴即理解"的语义**：粘贴的是一段 Markdown 文本，不是一个结构化的 Context Bundle——AI 需要额外的指令来理解"这是一个 Context Bundle，请据此理解之前的讨论"

**严重度**：★★★★☆

**改进建议**：

- **P1**：在 Context Bundle 的头部添加一段 AI 可理解的"导入指令"——例如：`<!-- The following is a CtxPort Context Bundle containing a previous AI conversation. Please read and understand this context before proceeding. -->`
- **P2**（长期）：提供 CLI 工具 `ctxport paste`，从剪贴板读取 Bundle 并格式化传递给 Claude Code / Cursor

---

### 4.2 断裂点 B2：AI 聊天 -> 另一个 AI 聊天

**断裂描述**：

从 ChatGPT 复制 Context Bundle，粘贴到 Claude.ai 的新会话中。

**断裂表现**：

- Claude.ai 的输入框有字符限制，长 Bundle 可能被截断
- 粘贴的 Bundle 被 AI 视为"用户的一条消息"——AI 不理解这是"之前在另一个 AI 中的多轮对话"
- 角色标注可能造成混淆：Bundle 中的"## Assistant"回复被 Claude 理解为 ChatGPT 的回复，还是自己应该扮演的角色？
- **上下文损耗**：原始对话中的隐式共识（"我们之前讨论过要用 REST 而非 GraphQL"）在 Bundle 中可能不够显式

**严重度**：★★★★★（这是 CtxPort 核心价值场景中最大的断裂）

**改进建议**：

- **P0**：Context Bundle 格式中应包含明确的元数据和角色标注，让目标 AI 理解"这是从 ChatGPT 导入的会话上下文"
- **P1**：在 Bundle 头部自动附加一段"上下文导入 prompt"——告诉目标 AI 如何解读这个 Bundle
- **P2**：研究不同 AI 对粘贴内容的最大长度限制，在复制前提示用户"此 Bundle ~30K tokens，Claude.ai 建议单条消息 <25K tokens，是否自动精简？"

---

### 4.3 断裂点 B3：非聊天平台 -> AI 工具

**断裂描述**：

从 GitHub Issues/PR、Stack Overflow、技术博客等非聊天平台复制内容到 AI 工具。

**断裂表现**：

- 这些平台不是 CtxPort 的原生支持平台（MVP 仅支持 AI 聊天平台）
- 用户经常需要"GitHub Issue 的讨论 + ChatGPT 的方案分析 + 自己的代码"组合成一个上下文
- 不同来源的内容格式不统一

**严重度**：★★★☆☆（MVP 可以暂不覆盖，但应在路线图中）

**改进建议**：

- **P3**（非 MVP）：提供"通用剪贴板收集器"——用户在任何网页上选中内容，右键选择"添加到 CtxPort Bundle"，多次收集后一键打包
- **P3**（非 MVP）：支持 GitHub 作为数据源（已在 plugin 列表中，可利用 `packages/core-plugins/src/plugins/github/plugin.ts`）

---

## 5. 新场景的交互设计建议

### 5.1 Vibecoding 场景：非技术用户的 Context 管理

**场景描述**：Maya 在 Cursor 中 vibecoding 一个 SaaS 产品，context window 即将耗尽。她需要把当前的项目状态"打包"到新会话中继续。

**交互设计建议**：

**核心原则**：Maya 不知道什么是 "token"、"context window"、"Markdown"。设计必须用她理解的语言和隐喻。

1. **"项目快照"隐喻取代"Context Bundle"**：
   - 对 Maya 展示为"保存项目快照"而非"复制 Context Bundle"
   - 图标使用相机/快照图标而非剪贴板图标
   - Toast 消息："已保存项目快照（含 24 条对话记录）"

2. **"容量指示器"取代 Token 数字**：
   ```
   ┌────────────────────────────────┐
   │  项目快照大小：████████░░  80%  │
   │  适合：Claude ✓  ChatGPT ✓     │
   │  可能过大：Cursor ⚠            │
   └────────────────────────────────┘
   ```

3. **"智能精简"一键操作**：
   - 当 Bundle 过大时，提供"自动精简"按钮——AI 辅助提取关键上下文，移除冗余对话
   - 显示精简前后的大小对比

---

### 5.2 项目级 Context 管理

**场景描述**：Alex 在一个持续 3 个月的项目中，分散在 ChatGPT（架构讨论 x 15）、Claude（代码审查 x 20）、Cursor（编码会话 x 50+）中积累了大量 AI 上下文。

**交互设计建议**：

**核心原则**：会话是资产，不是消耗品。项目级管理需要"仪表盘"而非"列表"。

1. **Extension Popup 作为"项目仪表盘"**（非 MVP，路线图功能）：
   ```
   ┌─────────────────────────────────┐
   │  CtxPort | 项目：API 重构        │
   ├─────────────────────────────────┤
   │  ChatGPT:  8 个会话  ~45K tokens│
   │  Claude:   5 个会话  ~32K tokens│
   │  ─────────────────────────       │
   │  合计: 13 个会话  ~77K tokens    │
   │                                  │
   │  [打包项目上下文]  [管理会话]     │
   └─────────────────────────────────┘
   ```

2. **自动项目关联**：
   - 基于会话标题关键词自动建议"这个会话属于哪个项目"
   - 用户确认后，后续复制自动附加项目上下文

3. **"项目上下文模板"**：
   - 为每个项目维护一个"项目背景 prompt"，每次复制时自动附加
   - 类似 CLAUDE.md 的作用，但跨平台通用

---

### 5.3 实时协作场景：团队 AI 上下文分享

**场景描述**：Leo 的团队 3 人同时在用不同 AI 工具讨论同一个技术方案。Leo 需要看到所有人的讨论并做出决策。

**交互设计建议**：

**核心原则**：分享应该是"推送"而非"拉取"。Leo 不应该去找团队成员要 AI 对话——团队成员应该能主动推送。

1. **"分享 Bundle"功能**（非 MVP，路线图功能）：
   - 复制 Context Bundle 后，toast 中增加"[分享]"按钮
   - 点击后生成一个临时链接（本地服务器 or P2P，不经过云端）
   - 链接接收者可以一键导入到自己的 AI 工具中

2. **"Bundle 收件箱"**：
   - Extension Popup 中增加"收到的 Bundle"标签页
   - 团队成员分享的 Bundle 在这里汇集
   - Leo 可以选择多个 Bundle 合并后投喂给自己的 AI

3. **MVP 可行的简化版**：
   - 复制 Bundle 后，用户可以选择"复制为可分享链接"——将 Bundle 编码为 base64 URL（适合小型 Bundle）
   - 或者"保存为 .md 文件"——通过 Slack/Email 发送

## 7. 总结：三个核心交互洞察

### 洞察 1：复制不是终点，"被理解"才是

当前设计把"写入剪贴板"视为成功——但从 Alex 的 End Goal 看，真正的成功是"目标 AI 理解了我之前的上下文"。从"复制成功"到"被理解"之间还有一段被忽视的路程：Bundle 的结构是否对目标 AI 友好？大小是否合适？角色标注是否清晰？这段路程上的每一个摩擦点都会导致用户放弃 CtxPort 而回到手动复制粘贴。

**设计行动**：Context Bundle 的格式设计应该以"目标 AI 的理解质量"为验收标准，而不是"剪贴板内容的完整性"。

### 洞察 2：Vibecoder 需要的不是"更多选项"，而是"更少决策"

当前设计的渐进式披露策略（默认一键复制，高级选项在右键菜单中）对 Alex 非常好。但对 Maya 来说，即使是"一键复制"也可能引起焦虑——"我复制了全部对话，但新 AI 能处理这么多内容吗？"Maya 需要的是**更智能的默认行为**：工具应该自动判断"这个 Bundle 是否适合目标工具"，而不是让 Maya 自己判断。

**设计行动**：在 Token 估算旁增加"目标适配指示"，让 Maya 一眼就知道"可以放心粘贴"还是"需要精简"。

### 洞察 3：团队协作是 CtxPort 从"工具"进化为"平台"的关键

MVP 聚焦个人使用是正确的。但 Leo 的痛点揭示了一个更大的市场：**团队级 AI 上下文管理**。当一个团队的 3-8 个成员各自与 AI 对话时，团队的"集体智慧"被锁在各自的会话历史中。如果 CtxPort 能让 Bundle 在团队间流动，它就不再是一个"复制工具"，而是一个"AI 知识协作平台"。

**设计行动**：在 MVP 后的第一个大版本中，引入最简的"分享 Bundle"能力——哪怕只是"保存为文件 → 通过 Slack 发送"的手动流程。
